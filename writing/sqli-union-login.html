SQLi via UNION in a custom login form

A methodical walkthrough of how I went from a 403 to a full authentication bypass in a deliberately vulnerable Flask app — payloads, mistakes, defenses, and what I'd do differently now.

8 min read Web · SQLi · Auth Published 18 Jun 2026
sqli union-based authentication flask owasp-a03

TL;DR

A custom login endpoint took username and password as POST form fields and concatenated them into a raw SQL string. A single ' OR '1'='1' -- in the username field returned the first row in the users table and let me log in as that user. From there, a UNION SELECT revealed the password column layout and showed me every account in plaintext. Total time from "this looks weird" to authenticated session: under fifteen minutes.

1. The setup

The target was a deliberately vulnerable Flask app I built for my own practice — the Vulnerable Login Lab in the Projects section. It exposes three routes: /login (GET form, POST submit), /dashboard (auth-only), and /admin (role-gated).

The relevant handler — note the string concatenation:

@app.post("/login")
def login():
    username = request.form.get("username", "")
    password = request.form.get("password", "")
    query = (
        "SELECT id, username, password, role FROM users "
        f"WHERE username = '{username}' AND password = '{password}'"
    )
    row = db.execute(query).fetchone()
    if row:
        session["user"] = row["username"]
        session["role"] = row["role"]
        return redirect("/dashboard")
    return "Invalid credentials", 403

What you should never see in production: user input going directly into a SQL string. Any value containing a quote can break out of the literal and inject arbitrary SQL. This is OWASP A03 — Injection, the single most common web vulnerability class for over a decade.

2. The first probe

I always start with the most basic test: send a single quote and see what happens. If the server returns a 500 with a SQL error trace, I know concatenation is happening. If it returns a 400/403/200 with a generic error, the app might still be vulnerable but is swallowing the exception. 

POST /login HTTP/1.1
Content-Type: application/x-www-form-urlencoded

username=admin'&password=anything

Response: 500 Internal Server Error with a SQLite traceback mentioning unrecognized token — confirming string concatenation in the query. Time to escalate.

3. Authentication bypass — the classic

The "bypass auth" test. Goal: make the WHERE clause evaluate true for some row, regardless of what I type in password. 

username=admin' OR '1'='1' --&password=anything

What the server ends up executing: 

SELECT id, username, password, role FROM users
WHERE username = 'admin' OR '1'='1' --' AND password = 'anything'

The first condition matches admin directly. Even if it didn't, '1'='1' is always true. The -- comments out the rest of the line, including the password check. The first row comes back; I get logged in as whoever is first in the table.

Response: 302 Found, Set-Cookie: session=.... I followed the redirect and was looking at the dashboard. Auth bypassed in two requests.

Why this worked

  • Input was concatenated raw, not parameterized.
  • The error path leaked the SQL flavor (SQLite), confirming quote handling.
  • There was no rate limiting, no WAF, no auth-failure alerting.
  • The query returned the full row on success — no per-field gating.

4. Going further: UNION-based extraction

Bypass is fun, but the real prize is data. The query returns id, username, password, role — four columns. A UNION SELECT with matching column count lets me fabricate a row containing anything I want, which the app will then echo back to me (via the dashboard's "welcome, {username}" line).

First, find the number of columns (4, in this case):

username=' UNION SELECT 1,2,3,4 --&password=x

Response: dashboard shows welcome, 2 — column 2 (the username slot) is reflected in the page. Now extract arbitrary data: 

username=' UNION SELECT 1, group_concat(username), group_concat(password), group_concat(role) FROM users --&password=x

Response: every user, every password hash (which in this lab is plaintext — a second vulnerability, but that's a different writeup), every role, in one screen.

Defender's note: UNION-based extraction only works when the original query's result is reflected somewhere the attacker can read. In this app, the username was echoed in the dashboard HTML. If the dashboard had shown a static "logged in" page with no user data, the UNION trick would still exfiltrate via blind techniques (boolean or time-based), just more slowly.

5. What I'd do differently now

  1. Test for blind variants first, not reflected ones. Reflected SQLi is rare in modern apps; time-based is much more common. My muscle memory is still calibrated to the easy case.
  2. Check for second-order SQLi: even if /login parameterizes input, the username gets stored and re-used in another query elsewhere (e.g., audit log, profile lookup). The dashboard reflection here is essentially second-order; in real apps the chain is longer.
  3. Map the database schema before dumping. sqlite_master / information_schema first, then targeted UNION SELECT. Dumping everything is noisy and the dashboard truncation can hide column count mismatches.
  4. Document the fix alongside the bug. The lab repo now ships a "fixed" branch using parameterized queries, and the README has a side-by-side comparison. That's what makes the writeup useful to other learners, not just the exploit.

6. The fix (one-line)

Parameterized queries. The vulnerable version above becomes: 

row = db.execute(
    "SELECT id, username, password, role FROM users "
    "WHERE username = ? AND password = ?",
    (username, password),
).fetchone()

The driver handles escaping. The input is data, never code. The exact same payloads return 403 "Invalid credentials" with no error trace.

7. Takeaways

  • String concatenation + user input is the original sin. It's been the #1 web vulnerability for twenty years for a reason.
  • Single-quote probe is still the right first move. Even when apps swallow the error, the latency and response size differences are usually detectable.
  • UNION extraction only needs column count match — find it with ORDER BY or sequential UNION SELECT 1,2,3....
  • Defense is one line of code; exploitation is two requests. The asymmetry is the lesson.
